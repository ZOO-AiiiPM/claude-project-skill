# lessons/ 写作标准

项目内**叙事性复盘**的专用目录。每份 lesson 是一个完整案例：起点问题、走过的弯路、最终方案、提炼规则。和 journal / rules / memory 的区别是：**lesson 讲故事，带原因和反转**，不是简短的事实或规则。

---

## 应该长什么样

### 目录结构

```
lessons/
├── memory-system-layout.md    # 一次重构的完整过程
├── api-migration-v2.md        # 一次迁移的踩坑故事
└── .gitkeep
```

扁平，文件名用 kebab-case 主题，不带日期（内容里写）。

### 文件结构

```markdown
# Lesson: {一句话主题}

{日期} {场景简述}。下次遇到类似 {具体情境} 前读一遍。

---

## 起点：{初始问题描述}

{背景 / 真问题是什么}

---

## 走过的弯路

### 坑 1：{简短标题}

**现象**：{观察到的症状}
**根因**：{真实原因}
**教训**：{下次怎么避免}

### 坑 2 / 坑 3 ...

---

## 最终架构 / 方案

{图 / 结构 / 代码片段}

---

## 可复用规则（提炼后）

1. **{命令式规则}**：{why}
2. ...

---

## 用户的自我观察（可选）

{本次协作里发现的自己 / agent 的模式}
```

### 长度约束

- **目标 3-8 KB / 100-300 行**
- 太短（< 1 KB）说明该进 journal 而不是 lesson；太长（> 10 KB）说明该拆成多份或压缩

### 触发阈值

不是每次踩坑都写 lesson。值得写的触发器：

1. **多步反转**：方案 A → 失败 → 方案 B → 发现问题 → 方案 C 成功
2. **反直觉结论**：最后做法和第一直觉完全相反
3. **跨领域串联**：一个踩坑触发重新理解 2+ 个无关系统
4. **可复用抽象**：案例里能提炼出 3+ 条命令式规则

不满足的情况下写 journal 的"坑了"字段就够，不要强行升级。

---

## 为什么这样

### 为什么 lesson 独立存在

journal 是**倒序短记录**，不适合叙事长反转。rules 是**命令式抽象**，丢掉了 why 和过程。memory 是**按需查的事实**，读者不知道什么时候查。

lesson 填补了这些：**按需读、带叙事、能让读者（包括未来自己和其他 Claude）重演推理过程**。没有过程，规则遇边界就失效；有了 lesson 做背景，规则失效时能回去看原始推理。

### 为什么要有"可复用规则"段

lesson 的终点不是故事，是**能抽成 rule 的规则**。每份 lesson 结尾列出"下次怎么做"的命令式条款，够硬的会被搬进 `.claude/rules/{主题}.md`。lesson 没有提炼段，等于只有故事没有产出。

### 为什么允许"用户的自我观察"段

协作模式是项目本身的一部分。用户在某次协作里发现自己"容易在边界处推翻 agent 的方案" / "需要被具体例子激活才能表达直觉" —— 这类观察对**下一次协作**价值极大，但不属于技术规则，单独放一段让它不被规则淹没。

## 判断标准（audit 时问的问题）

1. **蒸馏效率**：lesson 结尾有没有"可复用规则"段？有没有把规则抽到 `.claude/rules/` 去？
2. **该拆没拆**：单份 lesson 超过 500 行？（太长说明应该拆多份）
3. **该升没升**：lesson 里的规则已经在多个项目验证有效 → 该进全局 rules（`~/.claude/rules/`）
4. **该降没降**：某份 lesson 内容太单薄（< 1 KB，没反转、没规则提炼）→ 该合并进 journal
5. **命名可读性**：文件名能让别人 5 秒猜出主题吗？（`memory-system-layout.md` ✓；`case-1.md` ✗）
6. **内容 vs journal 重复**：同一事件在 journal 和 lesson 里重复写 —— 应删 journal 里的详版只留"详见 lessons/XXX"指针
7. **过期 lesson**：讲述的场景已经完全不存在（用的技术栈已废弃）→ archive

---

## 反模式

- **把 journal 当 lesson 用**：每天都新写 lesson，结果是一堆流水账而非反思。lesson 按主题不按时间
- **lesson 里只讲故事不提炼规则**：看完感慨 "好反转" 但下次还是会再踩同样的坑
- **一份 lesson 塞多个主题**："我这周遇到的几件事" —— 拆
- **lesson 里写代码教程**：教程进 docs/，lesson 是反思不是手册
- **deprecated lesson 留着**：技术栈换了，lesson 讲的场景不复存在 → archive/（不删，作为历史）

## 走过的弯路

**现象**：... **根因**：... **教训**：...

## 最终架构

## 蒸馏链（lesson 在链里的位置）

```
实战踩坑
  ↓ 一句话
journal 的"坑了"字段
  ↓ 同类重复 2+ 次，或单次反转复杂
lessons/{主题}.md（叙事 + 提炼规则）
  ↓ 规则稳定 + 有复用价值
.claude/rules/{主题}.md（命令式）
  ↓ 跨项目都适用
~/.claude/rules/{主题}.md（全局）
```

lesson 是链中的"半成品"：比 journal 深，比 rules 软。软意味着能保留 why 和反转；能升级到 rules 时就升级。

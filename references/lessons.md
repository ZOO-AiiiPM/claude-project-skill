# lessons/ 写作标准

`lessons/` 是**蒸馏链的半成品**：比 journal 的短反思深一层，比 rules 的命令式抽象软一层。在链条里它的位置是 journal → **lesson** → rules → 全局 rules，填补的是 "一句话反思" 和 "命令式规则" 之间的空白。

没有 lesson 的项目会出现两种退化：journal 里只有 "坑了：X 失败" 一句话，信息太少下次还是重蹈覆辙；rules 里是命令式规则，但丢失了"为什么这样做"和"曾经走过什么弯路"。lesson 填这个空 —— 它是**带叙事、带反转、带 why 的完整案例**，让读者（未来的自己和其他 Claude）能重演原始推理过程，而不只是知道结论。

---

## 不是每次踩坑都写 lesson

写不写 lesson 有门槛 —— 这个门槛不高，但也不是每条"坑了"都要升级。触发写 lesson 的信号四种：

**多步反转**：方案 A → 失败 → 方案 B → 发现新问题 → 方案 C 才成功。这种完整故事 journal 的三段式装不下，需要 lesson 的叙事结构来展示"为什么 A 不行、为什么 B 也不行、最后为什么 C 能通"。

**反直觉结论**：最后做法和第一直觉完全相反。这种 lesson 的价值在"记录直觉错在哪" —— 下次遇到类似情境时能提醒自己绕开直觉路径。

**跨领域串联**：一个踩坑触发了对 2+ 个无关系统的重新理解。这类案例讲"坑"没意义，讲"这坑让我看清了什么结构"才是价值所在。

**可复用抽象**：案例里能明确提炼出 3+ 条命令式规则。三条以上的规则集中在同一主题，值得用 lesson 做背景载体，让规则和故事一起留档。

不满足任何一条的踩坑，journal 的"坑了"字段就够，不要强行升级。你可能觉得"仪式感地写 lesson 显得重视" —— 但凑数 lesson 会稀释真 lesson 的价值，下次翻 lessons/ 时要筛掉凑数的才能看到重点案例。lesson 的数量应该少而精。

---

## 必须有"可复用规则"段

lesson 的终点不是故事，是**能抽成规则的认知**。每份 lesson 结尾必须有一段 "可复用规则"，把故事里学到的东西写成命令式清单。

```markdown
## 可复用规则（提炼后）

1. **读缓存必配写失效**：update / delete 后必须删缓存，否则就是在制造不一致
2. **热 key 需 single-flight**：爆款 key 必须加互斥，避免 stampede
3. **TTL 要加随机抖动**：固定 TTL 会让大量同时生成的 key 集体过期，集体打穿
```

没这段的 lesson 是"感慨式笔记" —— 看完觉得"好反转"，下次还踩同样的坑。有这段之后，规则稳定有效时能直接搬进 `.claude/rules/{主题}.md`（规则 + 触发锚点 + 场景 paths），lesson 本体留作案例背景。这是 lesson 升级到 rules 的机制。

你可能觉得写完故事就差不多了 —— 但故事不留规则等于没产出。强制结尾段是 lesson 唯一的产出保证。

---

## 结构和长度

推荐结构（可选段按需加）：

```markdown
# Lesson: {一句话主题}

{日期} {场景简述}。下次遇到 {具体情境} 前读一遍。

## 起点：{初始问题描述}

{背景 / 真问题是什么}

## 走过的弯路

### 坑 1：{简短标题}
**现象** / **根因** / **教训**

### 坑 2 / 坑 3 ...

## 最终架构 / 方案

{图 / 结构 / 代码片段}

## 可复用规则（提炼后，必须）

1. **{命令式}**：{why}

## 用户的自我观察（可选）

{协作里发现的自己 / agent 的模式}
```

"用户的自我观察"是软选段 —— 协作模式是项目的一部分，"我容易在边界处推翻 agent 方案"、"我需要被具体例子激活才能表达直觉"这类观察对下次协作极有价值，但不属于技术规则，单独一段让它不被规则淹没。

长度参考：目标 100-300 行 / 3-8 KB。太短（< 1 KB）说明内容还没撑起 lesson，应该回去写 journal；太长（> 500 行）说明塞了多个主题，应该拆。这不是硬阈值而是**内容自然边界的指标** —— 太短意味着故事没讲完整（反转不够、规则没提炼够），太长意味着主题不聚焦。

## 命名和组织

扁平目录，文件名用 kebab-case 主题，不带日期。日期写在文件内容里，不进文件名。

```
lessons/
├── cache-stampede-recovery.md     # 主题清晰
├── auth-migration-v2.md           # 能 5 秒看懂
└── memory-system-redesign.md
```

反面命名：`case-1.md`（主题不明）、`2026-05-02-debug.md`（带日期，后来归档也不好处理）、`my-lesson.md`（无信息）。

"5 秒看懂主题" 是命名的检验标准 —— 文件名用户 / 未来自己 / 其他 Claude 扫一眼就知道该不该读，不需要打开内容。

## 审视 lessons/

核心判断：**每份 lesson 的蒸馏过程是否完整、是否卡在某一环**。蒸馏完成 = 有故事 + 有规则 + 规则升级到 rules；卡住的每一步都是要改的地方。

具体看：lesson 没"可复用规则"段 → 蒸馏半途而废，补提炼段或降回 journal。"可复用规则"段写了但从未搬进 `.claude/rules/` → 蒸馏链下一步没走，搬过去。单份超过 500 行 → 拆多份。单份不到 1 KB 没反转没规则 → 降回 journal。命名 5 秒看不懂主题 → 改名。某事件在 journal 和 lesson 里重复详写 → journal 里留一行指针 "详见 lessons/XXX"，详版只留 lesson 一处。讲述的场景已完全不存在（技术栈废弃）→ archive/（不删作为历史）。lesson 里的规则已在多个项目验证有效 → 该升到全局 `~/.claude/rules/`。

共同根因：lesson 的健康 = 蒸馏链是否在流动。流动正常：踩坑 → journal 一句话 → 重复或反转升级 → lesson → 规则抽出到 rules → 有的升全局。任一环卡住都是信号：journal 一堆重复"坑了"但没升级 lesson = 蒸馏起点没启动；lesson 堆着没抽规则 = 中间环节卡住；rules 从没加过条目 = 链条末端不通。审视 lessons/ 本质是审视蒸馏链。

## 反面长什么样

最常见的失败是**把 lesson 当别的文件用**，共同根因是没想清 lesson 在蒸馏链里的位置（叙事 + 规则提炼）。

**当 journal 用**：每天新写一份 lesson。结果是一堆流水账而非反思。lesson 按主题不按时间，同一主题下多次反转算一份，不是一次反转一份。

**只讲故事不提炼规则**：文章结构完整（起点 + 弯路 + 方案）但没"可复用规则"段 —— 读完感慨 "好反转" 但下次还踩。这是 lesson 的最高频失败模式。

**一份 lesson 塞多个主题**："我这周遇到的几件事"风格 —— 拆成多份，每份聚焦一主题。

**lesson 里写代码教程**：教程给人读进 docs，lesson 是反思不是手册。代码片段 OK（作为背景展示），但不要把 lesson 写成"怎么做 X 的分步指南"。

**deprecated lesson 留着占位**：技术栈换了、场景不复存在 → 迁 `archive/`（不删，保留历史）。留在主列表误导读者"这是当前场景"。

这些失败共同特征是**没把 lesson 当蒸馏链半成品看**。看成日记就变流水账；看成手册就失去反思；看成笔记就不提炼规则。想清楚它是"带叙事的规则原材料"，写法自然正。

## 示例

### 好的 lesson 骨架

```markdown
# Lesson: 商品详情页加缓存的 3 次反转

2026-05-02 给 `/products/:id` 加 Redis 缓存，上线后连踩 3 个坑的完整复盘。

## 起点：读多写少，想加缓存

详情页 QPS 过万，DB 扛不住。Redis 缓存 + TTL 30 分钟，第一版上线顺利。

## 走过的弯路

### 坑 1：库存改了，前端还是旧值

**现象**：运营后台改完库存，用户端 10+ 分钟后才看到新值
**根因**：只写了 read 逻辑（get 先查缓存），没写 write 逻辑（update 没 invalidate）
**教训**：缓存的读写必须成对，只做读不做写 = 在制造不一致

### 坑 2：爆款商品并发打穿

**现象**：某商品缓存失效瞬间，DB QPS 暴涨几千
**根因**：N 个请求同时发现缓存空，全打库并回写（cache stampede）
**教训**：热 key 需 single-flight 或概率性提前刷新

## 最终架构

{client → Redis → DB，write path 同步 delete cache，热 key 加互斥锁}

## 可复用规则（提炼后）

1. **读缓存必配写失效**：update / delete 后必须删缓存，否则制造不一致
2. **热 key 需 single-flight**：爆款 / 首页 / 榜单类必须加互斥，避免 stampede
3. **TTL 要加随机抖动**：大量同时生成的 key 用固定 TTL 会集体过期打穿

## 用户的自我观察

第一版只想 "把读加速"，没想 "写怎么办" → 下次做缓存设计时先画完整读写时序图再动手
```

反转清晰（3 坑串起来）、规则段明确（3 条命令式，每条带 why）、自我观察段捕捉了协作层规律。这份 lesson 的规则段如果再验证有效几次，可以搬进 `.claude/rules/caching.md`。

## 蒸馏链里的位置

lesson 不是终点，往前衔接 journal，往后升级到 rules：

journal 一句话"坑了" → 同类重复 2+ 次或单次反转复杂 → `lessons/{主题}.md`（叙事 + 提炼规则）→ 规则稳定 + 有复用价值 → `.claude/rules/{主题}.md`（命令式）→ 跨项目都适用 → `~/.claude/rules/{主题}.md`（全局）。

lesson 的"软"（保留 why + 反转）是链条需要的 —— 有了 lesson 做背景，规则在边界情况失效时能回去看原始推理；没有 lesson 只留 rules，规则就变成无根的命令。这是 lesson 不能被 rules 完全取代的原因。

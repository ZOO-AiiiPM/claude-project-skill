# .claude/rules/ 写作标准

项目规则文件。自动加载，给 Claude 稳定的行为约束。定位和 CLAUDE.md 的区别：**CLAUDE.md 是"每次都读"的顶层索引和短规则；rules/ 是"按主题拆分"的长规则，支持 `paths:` 作用域限定到特定文件类型触发**。

---

## 应该长什么样

### 目录结构

```
.claude/rules/
├── skill-writing.md        # 写 skill 时触发（paths: skills/**/*.md）
├── database-migration.md   # 改 migration 时触发
└── api-error-handling.md   # 全局触发（无 paths）
```

### 文件结构

```markdown
---
paths:
  - "skills/**/*.md"
  - "skills/**/*.py"
---

# {主题} {N} 条硬规则

违反会导致 X 问题。来自 {具体案例}。

## 1. 规则标题

规则正文：命令式、带触发锚点、解释 why。

## 2. ...
```

### Frontmatter

- **`paths:`**（可选）— glob 模式，只在匹配文件时触发。没写 = 全局加载
- 其他字段不需要

### 写作风格

- **命令式，不用柔化词**。"禁止 X" / "必须 Y" / "X 时用 Z"，不用 "尽量 / 可能 / 建议"
- **解释 why，不只是 what**。模型智能，知道原因能在边界情况自己推理；死规则遇边界会失败
- **示例即规则**。"请求 X 时参数必须为 Y" 不如直接给 `X(param=Y)` 示例，Claude 模仿最强的就是距离最近的同类示例
- **关闭错误路径比指向正确路径更有力**。"搜索无论搜多少次都拿不到这些数据" 比 "你应该调详情接口" 更有效 —— 不关错误路径，模型总会回去试

## 为什么这样

### 为什么独立 rules/ 而不是全写 CLAUDE.md

CLAUDE.md 每次启动都全量加载，每一行都在吃 context 预算。大部分规则（"写 X 时做 Y"）只在特定场景相关，塞进 CLAUDE.md 是 99% 对话的浪费。

`.claude/rules/{主题}.md` + `paths:` 作用域解决这件事：
- 规则还是自动加载（不用 Claude 主动读）
- 但只在相关场景加载（`paths:` 限定触发文件）
- 不相关场景不进 context

### 为什么按"主题"拆不是按"来源"拆

按主题（`skill-writing.md` / `database-migration.md`）Claude 能在相关任务开始时命中整组规则，形成完整行为约束。

按来源（"feedback_2026_04_24.md" / "fix_after_bug_xxx.md"）等于把规则按时间顺序打散，Claude 同一个任务要去拼多个文件才能凑齐规则。

### 为什么禁柔化词

"尽量" / "可能" / "建议" 对模型来说是**可忽略**的。遇到边界情况，带柔化词的规则会被当作"不重要的建议"跳过，带命令式的规则会被当作"硬约束"遵守。

### 全局 rules（`~/.claude/rules/`）vs 项目 rules（`.claude/rules/`）

- **全局**：跨项目稳定偏好（工具使用 / 记忆管理 / 环境约束）。本项目不重复写
- **项目**：项目特有的（API 规则 / Skill 写作 / 业务约束）。全局 + 项目自动合并加载

## 判断标准（audit 时问的问题）

1. **命令式 vs 原则式**：每条规则都是"做 X / 不做 Y"吗？还是"原则上应该考虑..."这种可忽略软建议？
2. **有触发锚点**：规则里标明 "X 时 / Y 场景" 吗？没锚点的规则等于空气
3. **解释 why**：每条规则给了 why 吗？只有 what 的规则在边界情况下失效
4. **示例即规则**：关键参数 / 写法有具体示例吗？还是只靠规则文字描述
5. **paths 合理性**：有 paths 的规则作用域对吗？全局规则放没 paths 的文件对吗？
6. **和全局 rules 重复**：`~/.claude/rules/` 已经有的规则，项目 rules 里在重复吗？应删项目侧保留全局
7. **命名 vs 实际内容**：文件名和内容匹配吗？（`api-rules.md` 里写了数据库规则 = 需要拆）

---

## 反模式

- **把 memory 当 rules 用**：memory 是事实（地址 / ID / 字段）；规则（做法 / 禁止）应该进 rules
- **把 journal 当 rules 用**：journal 是"做了什么"叙事；规则是抽象出的行为约束
- **一个 rule 文件塞所有规则**：`rules.md` 含 skill / API / 数据库全部规则 —— `paths:` 作用域就失效了，应按主题拆
- **复制全局 rules 到项目 rules**：全局已有的不用项目复述，反而冲突时难协调
- **用 `feedback_*.md` 代替 rules**（已废弃的 memory 类型）：迁到 `.claude/rules/{主题}.md`
- **规则里全是 ALWAYS / NEVER 大写**：Skill 写作指南明确提示这是 yellow flag —— 与其拍桌子，不如解释 why

## 示例

### 好的 rule（带 paths 作用域）

```markdown
---
paths:
  - "skills/**/*.md"
  - "skills/**/*.py"
---

# Skill 编写 5 条硬规则

违反会让 skill 低命中或被 agent 误用。来自实战重构（sort_by 错误 4 次 → 0 次、schema 阅读率 0% → 67%）。

## 1. 写指令，不写文档

SKILL.md 主体必须指令风格：按执行顺序、只展示当前场景要用的参数、假设读者只看一遍。references/ 可以文档风格，SKILL.md 不行。

## 2. 在先验最强的地方下最大力气

反直觉程度决定解释力度。模型直觉对的一笔带过；直觉错的花 3 行解释**为什么**。
```

- `paths` 精准：只在写 skill 相关文件触发
- 每条规则有 why（"因为模型直觉对的一笔带过" 这种解释）
- 规则和反模式对照（给了 "别对所有参数写一样长的说明" 这种反面指引）

### 坏的 rule

```markdown
# 项目规则

- 代码应该清晰
- 测试覆盖率尽量高
- API 调用要考虑错误处理
- 最好使用最新的库版本
```

- 没 frontmatter 作用域 → 全局加载浪费预算
- 全是柔化词 → 无约束力
- 没 why → 边界失败
- 没示例 → Claude 不知道怎么落地

## 蒸馏来源

rules 不是凭空写。来源：

1. **用户重复纠正**：同类错误被纠正 2+ 次 → 抽成 rule
2. **严重事故**：一次错误成本高（丢数据、误删文件）→ 立即抽成 rule
3. **lesson 成熟**：lesson 里提炼的规则稳定有效 → 抽成 rule

journal 的"坑了"→ lesson 的案例 → rule 的命令式。rule 是蒸馏链终点。

# .claude/rules/ 写作标准

`.claude/rules/` 是**按场景触发的长规则**的家。它和 CLAUDE.md 的分工是：CLAUDE.md 每次 session 强加载，适合放短规则 + 索引；rules 支持 `paths:` 作用域（glob 模式），**只在匹配到相关文件时加载**，适合放长规则和场景化规则。这个"按场景加载"是 rules 区别于 CLAUDE.md 的本质 —— 让项目规则既有自动激活（不用 Claude 主动读），又不吃 99% 对话的预算。

理解这层分工：大部分项目规则（"写 skill 时必须 X"、"改 migration 时必须 Y"）只在特定任务相关。塞进 CLAUDE.md 意味着每次无关对话都要加载这些规则，纯浪费；不写进任何自动加载文件又意味着 Claude 做相关任务时不会遵守规则。`paths:` 解决这个两难 —— 规则还在自动加载，但只在 Claude 接触到 glob 匹配的文件时加载。

---

## 命令式，不用柔化词

"尽量" / "可能" / "建议" 对 Claude 等同于"可忽略" —— 遇到边界情况第一个被抛弃。"禁止" / "必须" / "X 时做 Z" 才会被当硬约束。"尽量使用清晰的命名" 模型读完当建议跳过；"布尔变量必须以 `is` / `has` / `can` 开头" 模型读完当规则遵守。

你可能觉得命令式太硬、不礼貌、不专业。但规则的价值在硬 —— 软规则等于没写。想讲礼貌就在规则后面解释 why，解释 why 让硬规则变得合理，不是用柔化词稀释规则本身。

对应地，大写的 `ALWAYS` / `NEVER` 也是反面 —— 这是 skill 写作指南里明确的 yellow flag。大写读起来像拍桌子，模型面对命令式规则已经会遵守，大写只在表达情绪上比平写多一点，成本是用户读起来像被吼。与其大写，不如解释 why；解释让规则能在边界情况里自己推理，大写不能。

---

## 带触发锚点，不然是空气

没有触发锚点的规则会被当空气。"代码要清晰" 模型看完不知道什么时候激活；"写 SQL 时 JOIN 必须用显式 `INNER JOIN` 而非逗号分隔" 模型在相关任务时会命中。

锚点有两层：一层是 frontmatter 的 `paths:`（文件级触发），一层是规则文本里的"X 时 / Y 场景"（任务级触发）。paths 精准限定到相关文件类型，规则文本再说清任务类型 —— 两层叠加让规则只在真正该触发的时候触发。

```yaml
---
paths:
  - "migrations/**/*.sql"
  - "migrations/**/*.py"
---

# 数据库迁移 3 条硬规则

违反会导致生产锁表或数据丢失。...
```

paths 不写等于全局加载，适合跨文件类型的通用规则（比如"API 错误处理规则"可能既影响后端 Python 又影响前端 TypeScript 调用方）。写 paths 是默认，不写 paths 是例外，选"不写"时要明确知道这条规则是真全局需要的。

---

## 解释 why 而不是只说 what

规则只给 what 不给 why，模型在边界情况会失效 —— 它遇到新情况没法推理"这算不算这条规则的管辖范围"。

举例：规则是"加 NOT NULL 列必须分两步"。只给 what，模型遇到"加带默认值的 NOT NULL 列"会困惑（这算不算分两步的场景？）；给 why（"加 NOT NULL 无默认值会全表扫 + 表锁大表直接 outage"），模型能推出"带默认值不全表扫，可以一步"。这不是模型在违反规则，是模型在**应用原理**到规则文本没覆盖的新情况。

why 不用长篇论证，一句话原理够了。"因为 Postgres 全表扫会锁表" 比 "因为生产安全" 有效得多 —— 前者是机制解释，模型能推理；后者是口号，只能死记。

---

## 示例即规则

Claude 学得最快的是**距离最近的同类示例**。"请求 X 时参数必须为 Y" 比不上直接给 `X(param=Y)` 的示例；抽象描述"错误码 1001 时重试、1002 时报用户"比不上给一段伪代码展示两种路径。

规则文字讲清楚原理，示例展示落地 —— 两者都需要。只有文字 Claude 要自己推怎么写代码，大概率推偏；只有示例 Claude 不知道原理，变体场景应用不上。

---

## 关闭错误路径比指向正确路径更有力

"搜索接口无论搜多少次都拿不到这些数据" 比 "你应该调详情接口" 更有效。不关掉错误路径，模型总会在困难时回去试那条 —— 因为它的训练先验里"搜索接口能找东西"这种印象很强，只靠正面指令压不住。

这个原则的应用点：如果某个方案是**明确的死路**（调用会失败 / 数据根本不在那里 / 某个 API 永远返回空），与其说"该用 X"，不如直接写"Y 永远拿不到 Z，别再试"。前者让模型有退路（"X 不行就回去试 Y 吧"），后者断退路。

---

## 项目 rules vs 全局 rules

`~/.claude/rules/` 是全局（跨项目稳定偏好：工具使用 / 记忆管理 / 交互风格 / 环境约束），`.claude/rules/` 是本项目（项目特有：API 约定 / skill 写作 / 业务规则 / 数据库习惯）。两者自动合并加载。

你可能想在项目 rules 里重复全局已有的规则"以防忘记"。但重复等于风险 —— 两处写法如果不一致，Claude 面对冲突会随机跳到训练先验（而不是按其中一条规则走）。应只在项目侧写**项目特有**的东西，全局已有的不复述。

反过来，如果发现某条规则在多个项目都适用 → 从项目 rules 升到 `~/.claude/rules/`，并删除所有项目侧的复本。这是蒸馏链的最后一步，rules 的最高归宿。

---

## 规则从哪里来

rules 不是凭空写。三个来源：

**用户重复纠正**：同类错误被纠正 2+ 次，说明这不是一次性失误，而是模型的系统性倾向 —— 抽成规则避免再犯。单次纠正写进 journal 或 CLAUDE.md 短规则段就够，2+ 次才升级到 rules。

**严重事故**：哪怕只发生一次，错误成本高（丢数据 / 误删文件 / 生产事故 / 大额费用）的情况立即抽成规则。这类规则优先级最高，CLAUDE.md 项目硬规则段或 rules/ 都可以。

**lesson 成熟**：lesson 里提炼出的"可复用规则"段稳定有效 → 搬进 rules/ 作为命令式执行版本，lesson 留作背景故事。这是蒸馏链的正常流向。

原则：rules 是**蒸馏产物**，不是一拍脑袋的建议。没有来源的规则（"我觉得应该这样写"）大概率是通用经验，不是项目特有 —— 这类建议的家在全局 rules 或根本不该写，不是项目 rules。

---

## 审视一份 rules/

核心判断：**每条规则能在正确时机激活、且激活后能让模型做出正确判断**。不能的地方就是要改的地方。

具体怎么表现为"不能"？有柔化词（尽量 / 可能 / 建议）→ 规则失去约束力，改命令式。没触发锚点（paths 和规则文本里都没有"X 时"）→ 规则等于空气，补锚点。只有 what 没有 why → 边界情况失效，补一句机制解释。没示例 → 落地时模型要自己猜，补最小示例。paths 作用域错位（全局规则写了过窄的 paths、或窄场景规则没写 paths）→ 要么全局要么限定，别两头不靠。和 `~/.claude/rules/` 重复 → 删项目侧。文件名和内容不匹配（`api-rules.md` 里写了数据库规则）→ 拆分。出现 `feedback_*.md` 类旧命名的 memory 文件迁进 rules → 检查迁移完整性。**发现规则本身错时重构（删错的、留对的），不打补丁加新说明** —— 补丁累积会把规则集变成"规则的规则"的复杂系统，反而看不清什么是真规则。

共同根因：rules 健康取决于**是否把规则当"按场景触发的自动行为约束"而非"项目笔记"**。项目笔记的写法（我觉得怎样好、一些建议、大概应该）放到 rules 里就失效 —— 它们不是规则，是建议，应该走 lesson 或 journal。

---

## 反面长什么样

最常见的失败是**把 rules 当别的文件用**。共同根因是没把"规则"和"事实 / 叙事 / 建议"的边界想清楚。

**当 memory 用**：写地址 / 账号 / API 端点这类事实。事实的家是 memory，rules 是做法。

**当 journal 用**：写"做了什么"的时间线。时间线的家是 journal，rules 是抽象出的稳定约束。

**一个 rules 文件塞所有主题**（`rules.md` 包含 skill / API / 数据库全部规则）—— paths 作用域失效，写 skill 时要加载数据库规则，浪费预算。按主题拆文件，每份 paths 精准到对应场景。

**复制全局 rules 到项目 rules**：前面讲过，冲突风险 > 保险收益。项目侧只留项目特有。

**规则没来源**（不是纠正、不是事故、不是 lesson 成熟）：多半是通用经验，不属于项目 rules。写进项目 rules 会稀释真规则的可见性。

## 示例

### 好的 rule（带 paths 作用域）

```markdown
---
paths:
  - "migrations/**/*.sql"
  - "migrations/**/*.py"
---

# 数据库迁移 3 条硬规则

违反会导致生产锁表或数据丢失。来自两次事故（一次全表扫锁死主库、一次 DROP 列后回滚失败）。

## 1. 加 NOT NULL 列必须分两步

Postgres 给已有大表加 `NOT NULL` 列且无默认值 = 全表扫 + 表锁，大表直接 outage。

**做法**：

```sql
-- Step 1（这次迁移）
ALTER TABLE users ADD COLUMN tier TEXT DEFAULT 'free';

-- Step 2（下次迁移，确认无回滚后）
ALTER TABLE users ALTER COLUMN tier SET NOT NULL;
```

## 2. DROP COLUMN 必须分两次发布

先停写（代码不再写这列 + 部署），等一个版本确认无回滚诉求，再真 DROP。回滚代码后如果列已 DROP，老代码启动即崩。
```

paths 精准只在迁移文件触发；每条规则给了 why（Postgres 机制 / 回滚崩溃机制）和示例（具体 SQL 片段）；来源（两次事故）说清楚规则出处不是凭空想。

### 坏的 rule

```markdown
# 项目规则

- 代码应该清晰
- 测试覆盖率尽量高
- API 调用要考虑错误处理
- 最好使用最新的库版本
```

没 frontmatter（全局加载浪费预算）；全柔化词（无约束力）；没 why（边界失效）；没示例（落地模糊）；没来源（通用建议，不该在项目 rules）。每一条单拎出来都是反面教材。

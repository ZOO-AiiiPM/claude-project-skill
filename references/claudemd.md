# CLAUDE.md 写作标准

项目根的 `CLAUDE.md` 是整套协作层里**唯一每次 session 都被强加载**的文件。这个特权决定了它和其他所有文件的关系：`.claude/memory/` 是按需查的事实、`.claude/rules/` 是按场景触发的长规则、`journal.md` 是 session 开头读顶部几条的时间线 —— 它们都有条件加载机制，只有 `CLAUDE.md` 无条件吃每次对话的 context 预算。所以它的本质不是"项目说明"也不是"规则手册"，而是**模型进入这个项目时必须先看到的极简协作骨架**。理解这一点，后面所有写法都能自己推出来。

---

## 只放规则，把事实赶出去

既然每行都在吃预算，那你往里塞的每一条信息都要通过一个问题：*下次启动时，没有这条是不是就做错事？* 是 → 留下；不是 → 赶走。

服务器 IP / API 地址 / 错误码表 / 账号 ID，这些在 99% 的对话里用不到，但写进来每次都要加载，纯浪费。它们的家是 `.claude/memory/reference_*.md`，那里有 MEMORY.md 索引做入口，模型真需要时会自己按描述命中并读。

项目做什么、怎么部署、架构图 —— 这是**给人读的**，家在根目录 `README.md` 或 `docs/`。写进 CLAUDE.md 会让模型加载一堆它不会用的背景知识，还会让你养成"在 CLAUDE.md 里找项目信息"的错觉（错误的，这个文件会被随时重构，不适合当文档）。

每次迭代做了什么、某个决策的过程、踩了什么坑 —— 这是**时间线**，家在 `journal.md`。写进 CLAUDE.md 等于把 changelog 每行都全量加载，而模型需要时会自己去读 journal（不在 CLAUDE.md 里复制）。

你可能觉得"顺手写进来方便"，但这个"方便"的代价是每次对话、包括完全不相关的对话，模型都要读一遍这些过期内容 —— **现在多写一行，未来每次 session 都在付这行的税**。

---

## 写了也要能被当规则看

把事实赶出去后，留下来的是规则。但"是规则"不等于"有约束力"，下面三个细节决定模型会不会当真。

**命令式，不用柔化词**（完整论述见 [rules.md](rules.md)）。"尽量使用 TypeScript" 模型会当建议跳过；"写 TypeScript 必须启用 `strict: true`" 模型会当硬约束遵守。CLAUDE.md 是每次加载的骨架，最容不得柔化词 —— 一条软规则每次都占对话预算却无约束力，净亏损。

**带触发锚点**。没有锚点的规则会被当空气。"代码要清晰" 模型看完不知道什么时候该激活；"改 API 时先读 `.claude/memory/reference_api_errors.md` 确认错误码" 模型在任务匹配时会命中。每条规则问自己：*这条什么时候该被激活？* 把激活条件写进规则本身。

**给结论但说清楚为什么**。规则只说 what 不说 why，模型在边界情况会失效 —— 它知道要做 X 但不知道什么算 X 的变体。解释 why 让模型在新情况里能推理"这个场景属于这条规则的哪个面"，而不是死记一条条具体命令。why 不用长篇论证，一句话够 —— "外部 API 调用必须设超时，因为无超时的挂起请求会占满 worker 线程" 比 "API 调用要设超时" 强得多。

---

## 必要的段，不多不少

一份活的 CLAUDE.md 长这样：**一句话项目描述 → 跨 Session 协作的硬规则（journal / memory / rules 分工）→ 项目本身的硬规则 → 关键资源索引 → 待做**。其他段都是冗余。

"跨 Session 协作"段最容易被省 —— 用户觉得"反正我知道怎么用"。但这段不是给用户看的，是给**模型**看的：没有它，模型不知道 session 开头该去读 journal 顶部、不知道新事实该写 memory、不知道用户纠正该进 rules。这一段沉寂，整个协作层就沉寂。

"项目硬规则"段随项目成长，短的留 CLAUDE.md（1-3 行），长的迁 `.claude/rules/{主题}.md`。判断标准不是行数，是"这条规则将来可能长吗" —— 可能长就直接起 rules 文件，不要等它在 CLAUDE.md 里膨胀到不得不拆。

"关键资源索引"段指向所有其他文件。这里的路径必须真实存在，否则模型按建议读会碰到死链接。审查时用 Glob 验证，不只看文本。

"待做"段是工作记忆，不是项目愿景 —— 完成即删，长期（30 天+）不动的"僵尸 todo"说明根本不会做，该删或迁到 journal 的决策里记录"搁置了因为 X"。

## 审视一份 CLAUDE.md

审视的本质不是"对照清单扫一遍"，而是**读完后问自己：模型看完这份文件，能不能立刻知道怎么在这个项目里协作**。答案为"能"是健康，答案为"不能"的地方就是要改的地方。

具体怎么体现"不能"？长度让你不自然地想翻页，多半塞了不该在这的东西 —— 找看起来像事实 / 日期 / 长篇解释的段，问它"每次启动都要吗"。有规则没触发锚点，模型读完不知道何时激活，等于空气 —— 给这些规则补"X 时 / Y 场景" 的触发词。有柔化词，模型当建议跳过 —— 换成命令式。索引的路径 Glob 不到，模型按建议读会失败 —— 修路径或删索引。占位符还在（`{PROJECT_NAME}` 这类），说明 init 完没人改 —— 改掉。

核心推理：每条内容通过"没这条模型会做错事吗"的检验，每条规则通过"模型知道何时激活且知道为什么"的检验。这两个检验通过，就算 100 行也健康；不通过，就算 30 行也不合格。

## 反面长什么样

最常见的失败模式是**把 CLAUDE.md 当别的东西写**。

当 README 写：塞项目背景、架构图、安装步骤。这些是给人读的，模型每次加载它们就是在浪费预算。搬到根目录 README 或 docs/。

当 changelog 写：堆"第 N 次迭代做了什么"。时间线的家是 journal，每条带日期的记录都该在那里倒序排列，不是这里。

当 cheat sheet 写：把服务器 IP / API URL / 错误码表堆进来当快查手册。快查手册的家是 `.claude/memory/reference_*.md`，那里有按需加载机制。

当笔记本写：把"上次踩的坑 + 解决方案"大段叙事写进来。短规则进 rules，复杂反转进 lessons，CLAUDE.md 不吸收叙事。

这些失败的共同根因是：**没把 CLAUDE.md 和其他文件的协作关系想清楚**。想清楚了，自然会知道每种内容的家在哪。

## 最小可用骨架

新项目起手时，CLAUDE.md 可以短到这样：

```markdown
# {项目名}

{一句话描述}

---

## 跨 Session 协作（硬规则）

四个位置各司其职：`CLAUDE.md` 放规则 + 索引每次加载；`.claude/rules/*.md` 放长规则自动加载；`journal.md` 放时间线，session 开头读顶部几条、里程碑 append 一条到顶；`.claude/memory/` 放事实，MEMORY.md 索引自动加载、主题文件按需读。

写入分层：事实进 `.claude/memory/{reference,project}_*.md`；用户纠正 / 严重错误短的进本文件硬规则段、长的进 `.claude/rules/{主题}.md`；进度 / 反思进 journal；复杂反转进 lessons。

---

## 项目硬规则

（按项目填）

---

## 关键资源索引

- 事实：`.claude/memory/MEMORY.md`
- 规则：`.claude/rules/*.md`
- 案例：`lessons/`
- 文档：`docs/`

---

## 待做

- [ ]（按项目填）
```

项目跑起来后：协作段基本不变，项目硬规则随踩坑累积（短留、长迁 rules/），索引随 memory / rules 增加更新。这三段的成长本身就反映了项目的协作成熟度。

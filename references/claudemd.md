# CLAUDE.md 写作标准

项目根目录的 `CLAUDE.md` 是每次 Claude session 启动时**强制全量加载**的唯一文件。它的定位决定了其他所有记忆机制的边界：**这里只放"每次启动都必须知道"的规则和索引，其他东西都应该在别处按需读。**

---

## 应该长什么样

### 硬约束

- **行数上限 ≤ 80 行**。超过说明塞了不该在这的东西（事实 / 日期块 / 长规则）
- **必须含的段**：
  1. 项目一句话描述
  2. 跨 Session 协作硬规则（何时读/写 journal / memory / rules 的分工表）
  3. 项目硬规则（本项目特有的禁止 / 必须 / 做法）
  4. 关键资源索引（指向 memory / rules / lessons / docs）
  5. 待做清单（< 10 条，完成即删）

### 写作风格

- **命令式**："禁止 X" / "必须 Y" / "X 时做 Z"，不用 "建议" / "可能" / "尽量"
- **指令不解释**：规则本身就是结论。需要解释的 why 写在 lesson 或 rule 里，CLAUDE.md 只留结论
- **指令带锚点**：每条规则加上触发条件（"SSH 时" / "写 skill 时" / "上传文件时"），没锚点的规则会被当空气

---

## 为什么这样

CLAUDE.md 每次 session 都进 context，每多一行都在吃每次对话的预算。塞进去的事实（服务器 IP / API 地址 / 错误码）在 99% 的对话里用不到，属于浪费；塞进去的长篇论证让 Claude 读完还是不确定该怎么做，属于噪音。

**规则 vs 事实的判别公式**：

> 这条信息下次启动时每次都需要吗？
> - 每次都要 → 规则，留 CLAUDE.md
> - 只在相关任务时要 → 事实，进 `.claude/memory/`

不确定时默认进 memory —— memory 有索引，需要时 Claude 会主动读；CLAUDE.md 进了就永远占预算。

---

## 判断标准（audit 时问的问题）

Claude 审视一份 CLAUDE.md 时，按顺序问自己：

1. **行数**：超过 80 行吗？超过多少？
2. **段完整性**：有没有"跨 Session 协作"分工表？没有 = journal / memory / rules 会自然沉寂
3. **规则 vs 事实混入**：里面有没有"事实"（服务器 IP / 接口地址 / 错误码 / 账号 ID / 日期块）该迁到 memory 的？
4. **柔化词**：有没有 "可能 / 尽量 / 建议 / 大概率 / 一般来说"？这类词让规则失去约束力
5. **索引完整性**：关键资源索引提到的路径都真实存在吗？（Claude 要 Glob 验证，不只是看文本）
6. **待做时效**：待做清单里有没有已经完成但没划掉的？有没有 30 天以上的"僵尸 todo"？
7. **占位符残留**：`{PROJECT_NAME}` / `{一句话项目描述}` 这类模板占位还在？

审查时给出：哪些条款是"规则"该留、哪些是"事实"该迁、哪些是"废话"该删。要带原文引用（行号）。

## 反模式

- **当 README 写**：把"项目背景 / 架构图 / 怎么安装"塞进来。这些是人读的，进 `README.md` 或 `docs/`
- **当 changelog 写**：把每次迭代日期 + 做了什么堆进来。进 `journal.md`
- **当 cheat sheet 写**：把服务器 IP / API URL / 错误码表堆进来。进 `.claude/memory/reference_*.md`
- **当笔记本写**：把"上次踩的坑 + 解决方案"大段叙事堆进来。短的进 rules，复杂反转进 lessons
- **使用 `feedback_*.md` 存用户纠正**（已废弃）：短规则进 CLAUDE.md，长规则进 `.claude/rules/{主题}.md`

## 示例

### 好的分工表（硬规则段示例）

```markdown
## 跨 Session 协作（硬规则）

| 文件 | 存什么 | 何时读/写 |
|------|-------|----------|
| `CLAUDE.md` | 规则 + 索引（< 80 行）| 每次 session 自动加载 |
| `.claude/rules/*.md` | 长规则 / 按主题拆 / 可带 paths 作用域 | 自动加载 |
| `journal.md` | 时间线进度、反思、待办 | **Session 开头读顶 3 条**；里程碑 append 顶部 |
| `.claude/memory/` | 事实（服务器 / 账号 / API / 命令）| MEMORY.md 前 200 行自动加载 |
```

这段话让 Claude 知道四个文件各干啥、什么时候看，就能自动激活 journal / memory / rules。没这段，三件事都会沉寂。

### 坏的规则写法

```markdown
- 尽量使用清晰的命名约定
- 通常建议测试覆盖率达到 80% 以上
- 可以考虑使用 TypeScript 来增强类型安全
```

全是柔化词，没触发条件，Claude 不会当规则。换成：

```markdown
- 写 TypeScript 必须启用 `strict: true`，报错禁止用 `// @ts-ignore` 绕过
- 测试覆盖率目标 80%，PR 里低于这个阈值要在描述里说明原因
- 变量命名：布尔值以 `is/has/can` 开头
```

## 最小可用骨架

一个合规的 CLAUDE.md 最短可以是这样（~30 行）：

```markdown
# {项目名}

{一句话描述}

---

## 跨 Session 协作（硬规则）

| 文件 | 存什么 | 何时读/写 |
|------|-------|----------|
| CLAUDE.md | 规则 + 索引 | 每次加载 |
| journal.md | 时间线 | 开头读顶 3 条；里程碑 append |
| .claude/memory/ | 事实 | MEMORY.md 自动加载 |
| .claude/rules/ | 长规则 | 自动加载 |

---

## 项目硬规则

- （按项目填）

---

## 关键资源索引

- 事实：`.claude/memory/MEMORY.md`
- 规则：`.claude/rules/*.md`
- 案例：`lessons/`
- 文档：`docs/`

---

## 待做

- [ ] （按项目填）
```

项目跑起来后，三段会长起来：**跨 Session 协作** 基本不变，**项目硬规则** 随踩坑累积（短的留 CLAUDE.md、长的迁 rules/），**关键资源索引** 随 memory / rules 增加而更新。

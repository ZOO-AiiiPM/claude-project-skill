# journal.md 写作标准

`journal.md` 是项目时间线里**代码看不见的东西**的家：做了什么决策、为什么放弃方案 A、踩了什么坑、学到什么。git log 已经记录 diff（做了什么代码变动），journal 补的是"为什么这样做 / 过程中有什么反思"这层 —— 这才是下次打开项目时真正有用的信息。

journal 和其他协作层文件的分工：memory 是按需查的事实（不走时间），rules 是按场景触发的规则（不走时间），lesson 是长叙事（单个复杂反转），journal 独占"按时间顺序记录过程"这块。所以它的写法、触发机制、蒸馏方向都围绕"时间线"这个本质展开。

---

## 倒序，最新在顶

Claude 读 journal 的默认动作是读前 N 行。最重要的信息（项目最近状态）必须在最前，否则读了等于白读。正序（老在顶）让 Claude 要 `tail` 才能看到最新，大文件上这既慢又容易超 token，而且 `tail` 不是 Claude 的默认动作，大概率就不读了。

你可能觉得正序看起来像自然日记、时间顺序读起来舒服。对人类读者是对的，但 journal 的主要读者是**下一个 session 的 Claude** —— 它读 `head` 会默认看到最近发生了什么，能立刻进入项目状态。倒序是这个定位下的唯一合理选择。

## 三段式：做了 / 坑了 / 学到

每条 journal 用三段式结构，可选加一段"决策"：

```markdown
## 2026-05-02 登录切 refresh token

- **做了**：login 接口返回从单 token 改为 `{access, refresh}` 对。client 3 处调用点同步更新
- **坑了**：第一版 refresh 过期时间填的秒数，JWT 库要毫秒，所有 token 1 秒就失效。抓库 changelog 才看到单位
- **学到**：这个 JWT 库过期时间单位是毫秒，文档没明说。该写进 `.claude/memory/reference_jwt_library.md`
- **决策**：refresh 有效期 7 天 | 对齐团队其他产品 | 考虑过 30 天但安全组反对
```

三段各有独立价值：**做了** 是外部观察（项目完成了什么），给下一个 session 快速了解进度；**坑了** 是内部反思（哪里卡住 / 怎么卡住的），这是 journal 区别于 git log 的核心价值 —— git log 不会告诉你当时卡了几小时在一个单位换算上；**学到** 是蒸馏出的认知（下次换个方式做 / 这类问题该查 X），积累到够硬时会升级成 lesson 或 rule。

"决策"段按需加，记录"选了 A 不选 B、因为 X、替代方案是 Y"的三元组。这条信息将来回顾"为什么是这样"时极其有用 —— 没有它，重看项目时会疑惑"当时怎么想的"。

三段退化成一段（"今天做了登录相关的事，修了 bug，挺顺利"）就丢失全部价值：没做了的细节、没坑了的反思、没学到的蒸馏，等同于白写。

## 真实 > 完整

没实质进展的日子空着比凑数好。"今天 review 了代码"、"继续开发"这种条目存在只会稀释 journal 价值 —— 下次翻 journal 想找到"最近有什么决策 / 踩坑"时，要在一堆凑数条目里筛，反而看不清。

你可能觉得每天都写让 journal 看起来完整、有连续性。但完整性不是 journal 目标，**有信息密度**才是。一周里有两天真正写了有价值的条目，比一周七天每天凑数好。

## 何时写（硬规则，在 CLAUDE.md 里声明）

**session 开头读 journal 顶部 3 条**。这一条让 journal 被 Claude 主动激活 —— 不读不知道项目状态，读了自然带入历史。CLAUDE.md 的"跨 Session 协作"段要明确声明这条，否则 Claude 每次 session 都是白纸起步。

**里程碑 / 重要决策 / 严重踩坑后 append**。这一条让 journal 被写入者主动激活 —— 不 append 就变成一潭死水。关键触发词：里程碑（完成某个阶段）、决策（选了 X 不选 Y）、踩坑（花超预期时间才搞定）、学到（蒸馏出一条认知）。符合任一条件就 append，不用犹豫。

**session 结束时合并当日条目**。同一天多次 append 会产生多个 `## YYYY-MM-DD` 标题，session 结束时合并进一个，保持按日期聚合。

这三条是 journal 能持续活着的钩子。没有钩子的 journal 必然在 1-2 周内沉寂 —— 这不是自律问题，是机制问题。所以项目模板里应该默认装 `turn-reflect` 类 hook：每 N 轮自动让 Claude 判断是否该 append journal，把"记得写"从靠用户自律变成机制保证。

## 审视一份 journal

核心判断不是对照清单，而是问：**这份 journal 能不能让新来的 Claude / 未来的自己快速恢复项目状态和历史决策**。不能的地方就是要改的地方。

具体怎么表现为"不能"？顶部条目是 30 天前的 → journal 的写入钩子实际没生效，要检查 CLAUDE.md 的"session 开头读 / 里程碑 append"硬规则是否声明、hook 是否在跑。顶部日期不是最新的 → 条目插错位置，违反倒序。三段式退化成一段 → 失去区分价值，提醒拆回来。出现凑数条目（"今天 review 代码"）→ 稀释信息密度，建议删。最近 7-14 天的"坑了"字段有反复出现的词 → 蒸馏信号，该抽成 rule 或 lesson。底部囤了 30 天+ 没处理的 decision 或学到里的行动项 → 僵尸 todo，清或 archive。被当 changelog 用（堆长 diff / 代码块）→ 格式污染，这些内容进 docs 或 memory。

共同根因：journal 健康取决于**写入节奏是否和协作习惯绑定**。绑定成功（硬规则 + hook）→ 活；只靠"记得写"→ 必沉寂。审视 journal 时不止看内容，更要问"写入钩子生效了吗"。

## 蒸馏链：journal 不是终点

journal 是项目认知的**原材料**，不是终点。蒸馏方向：

journal 一句话"坑了" → 同类重复 2-3 次 → 升级成 `lessons/{主题}.md` 详细叙事 → 规律清晰到能写命令式 → 抽到 `.claude/rules/{主题}.md` → 跨项目适用 → 升到 `~/.claude/rules/{主题}.md` 全局。

不是每条"坑了"都要蒸馏。一次性的低价值踩坑留在 journal 就够，不升级 —— 蒸馏也有成本，单次坑写 lesson 是过度设计。蒸馏信号有两种：**重复**（同类坑 2+ 次说明系统性问题）、**反常规反转**（方案 A → B → C 的完整故事值得留档）。

审视 journal 时看"坑了"字段的重复模式，这是最有价值的蒸馏线索 —— 模型自己不会主动注意到，用户或审视者要主动找。

## 反面长什么样

最常见的失败是**把 journal 当别的文件用**。共同根因是没把"时间线反思"和"changelog / 事实 / 规则 / 教程"的边界想清楚。

**当 changelog 用**：堆"第 N 次迭代做了什么 + diff"。git log 已经有这部分，journal 复述就是浪费 —— journal 的价值是 diff 里看不到的"为什么 / 反思"。

**当事实仓库用**：在 journal 里长期维护 API 地址 / 错误码 / 配置。这些是事实（按需查），家在 memory。journal 保持"时间线反思"性质。

**当规则本用**：在 journal 里写"以后做 X 必须 Y"。规则的家是 rules 或 CLAUDE.md，journal 只记"今天发现了这条规则的苗头"。

**当代码仓库用**：journal 里贴大段代码块 / 长表格。代码进源码或 docs，journal 保持轻量反思。

**正序（老在顶）**：前面讲过，Claude 读 `head` 就看到远古信息，journal 失去价值。

**日期格式不统一**（"5/2 今天"、"5 月 2 日"、"2026-5-2"）：Claude 没法按日期可靠排序和审视，格式必须统一到 `YYYY-MM-DD`。

这些失败共同特征是**没想清 journal 的定位是"按时间顺序的短反思"**。想清楚了自然知道 changelog、事实、规则、代码、教程都该走别家。

## 示例

### 好的条目

前面已经给过一条（2026-05-02 登录切 refresh token）。重点看：三段都有信息密度，"学到"明确指出下一步（迁到 memory），"决策"带 why + 替代方案。

### 坏的条目

```markdown
## 5/2

今天继续做登录模块，修了一些 bug，晚上优化了下代码，挺顺利的。
```

日期格式不统一、无结构、没信息量、无行动项 —— 这种条目存在会让后来者读 journal 时把它跳过去找有信息的条目，浪费注意力。存在的危害大于不存在。

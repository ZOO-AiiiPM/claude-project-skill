# journal.md 写作标准

`journal.md` 是项目的**倒序时间线**。它不是 git log 的复述（diff 已经在那儿），而是那些"代码里看不出来"的东西：做了什么决策 / 为什么放弃方案 A / 踩了什么坑 / 学到什么。

---

## 应该长什么样

### 硬约束

- **倒序**：最新条目在顶部。看文件第一眼就是"最近发生了什么"
- **条目格式**：`## YYYY-MM-DD 标题`，三段式正文 `做了 / 坑了 / 学到`，可选 `决策`
- **真实 > 完整**：能写真话的写真话，没内容的日子不强写。假装每天都有收获比空着更糟

### 条目结构

```markdown
## 2026-05-02 实现 X 功能

- **做了**：具体做了什么（1-3 条，动词开头）
- **坑了**：踩了什么坑，怎么发现的（没有就写"-"）
- **学到**：下次该怎么做，或更新了哪条认知
- **决策**（可选）：选了 X 不选 Y | 为什么 | 替代方案是什么
```

### 何时写

**硬规则（在 CLAUDE.md 里声明）**：
- **Session 开头**：读 journal 顶部 3 条，快速了解项目最近状态
- **里程碑 / 重要决策 / 严重踩坑后**：append 一条到顶部
- **Session 结束**：如果今天有实质进展，合并进当天条目（同一天多条会被 append 到同一个 `## YYYY-MM-DD` 下）

不强制每天写。**没进展的日子跳过**，比凑数的"今天 review 了代码"好。

---

## 为什么这样

### 为什么倒序

Claude 读 journal 的默认动作是 `head` 或读前 N 行。**最重要的信息（最新状态）必须在前**，否则读了等于白读。正序的 journal 会让 Claude 不得不 `tail`，这在大文件上又慢又容易超 token。

### 为什么三段式

- **做了** = 外部观察（完成了什么），给自己和下一个 session 的 Claude 快速了解项目进度
- **坑了** = 内部反思（哪里卡住、怎么卡住的），这是 journal 相对 git log 的核心价值
- **学到** = 蒸馏出的认知（下次换个方式做 / 这类问题该查 X），这部分积累到够硬时会变成 lesson 或 rule

### 为什么硬规则激活

用户经常"建过 journal 但没用上"。根因是**没钩子**。"session 开头读顶 3 条" + "里程碑 append" 是最小可行钩子 —— 不写进 CLAUDE.md，journal 必然沉寂。

---

## 判断标准（audit 时问的问题）

1. **活跃度**：顶部条目是几天前的？7 天内 = 健康，8-30 天 = 需警惕，30+ 天 = journal 硬规则实际上没生效
2. **倒序正确性**：顶部日期是不是最新的？有没有条目插错位置？
3. **条目结构完整性**：每条都有"做了 / 坑了 / 学到"吗？还是退化成流水账？
4. **真实性**：有没有明显敷衍的条目（"今天继续开发"这种）？这种条目应该删或合并
5. **重复模式**：最近 7-14 天的"坑了"字段里，有没有**反复出现的词**？有 = 该蒸馏成 rule 或 lesson 了
6. **长期待办囤积**：journal 底部有没有超过 30 天没处理的 `决策` 或"学到"里的行动项？
7. **格式污染**：有没有被当 changelog / commit log / 代码片段仓库用？这类内容该迁走

审查给出：哪些条目该合并删掉，哪些"坑了"该蒸馏成 rule / lesson，journal 活跃度是不是正常。

---

## 反模式

- **正序（老 → 新）**：Claude 读顶部看到的是远古信息，失去 journal 价值
- **全部记录所有进展**：凑数日记。没实质进展的日子空着比写"今天 review 代码"好
- **把 git diff 复述进来**：journal 的价值是**diff 里看不到的东西**（为什么这么选、什么方案放弃了）
- **在 journal 里写长代码块 / 大表格**：这些进 docs/ 或 memory/，journal 保持"反思"性质
- **"做了 / 坑了 / 学到"退化成一个 "今天做了..."**：失去三段的区分价值
- **日期格式不统一**（"5/2 今天"、"5月2日"）：Claude 没法按日期排序，审查失败

---

## 蒸馏链

journal 不是终点，是**原材料**。蒸馏方向：

```
journal 一句话"坑了"
    ↓（连续 2-3 次同类坑）
lesson 详细叙事（复杂反转 / 多步踩坑的完整故事）
    ↓（规律清晰到能直接写规则）
.claude/rules/{主题}.md（命令式规则，以后自动避坑）
```

不是每条"坑了"都要蒸馏。一次性的低价值踩坑留在 journal 就够，不用升级。只有**重复出现**或**反常规反转**的才值得写 lesson，只有**已确定规律**的才值得写 rule。

## 示例

### 好的条目

```markdown
## 2026-05-02 搜索接口改走分页

- **做了**：search-library 接口改用分页参数（page/size），替代旧的 offset 方式。改了 scripts/kox.py 的 3 处调用点
- **坑了**：第一版 page 从 0 开始，线上测试 100% 返回空列表。抓 raw 才发现服务端 page 从 1 开始，文档没写
- **学到**：KOX 分页接口 page 从 1 开始，不是 0。这条该写进 `.claude/memory/reference_kox_api_rules.md` 的分页段
- **决策**：保留 size 默认 20 | 对齐前端默认值 | 考虑过 50 但怕 response 太大
```

三段都有信息含量，"学到"明确指出了下一步（迁到 memory），"决策"有 why 有替代方案。

### 坏的条目

```markdown
## 5/2

今天继续做搜索模块，修了一些 bug，晚上改了 skill，挺顺利的。
```

日期格式不统一、无结构、没信息量、无行动项。这种条目存在会稀释 journal 价值。

# 通用编码防坑 Preset

本文件是 `/project-setup init` 按项目类型装的**通用预设**，不是本项目特有规则。如果你已有 `andrej-karpathy-skills:karpathy-guidelines` 或类似全局 skill，本文件可删（全局 skill 触发更精准，重复反而稀释）。

保留本文件的场景：没有上述全局 skill 的项目 / 开源给别人用的项目（假设别人也没装）。

---

## 1. 先做最小改动，不要过度工程

改代码前问自己：**这个问题需要的最小改动是什么**。不是"怎么优雅"，是"最少能解决它的代码"。30 行能解决就不写 100 行，不要顺手重构周边（除非用户要求），不要预加"将来可能用到"的抽象。

过度改动的代价：审查难、回滚难、引入新 bug 的风险比修 bug 高。遇到"这里顺手也改下吧"的冲动，停下来问："用户要求改这个吗？" 没要求就不改。

## 2. 假设必须显式并验证

写代码时做的**假设**要在代码里显式化（注释、断言、日志）或通过工具验证。

- "假设这里一定是 list 而不是 None" → 加 `assert` 或显式判断
- "假设 API 返回 JSON" → 加错误分支处理非 JSON
- "假设并发写不会冲突" → 问数据库 / 缓存层有什么并发保证

假设不写出来，模型（和人类）都倾向忘记验证。边界情况真触发时 crash 的是假设未写的地方。

## 3. 成功标准必须可验证

写完代码问自己：**怎么知道它工作了**？

- 有单元测试 → 跑，看通过
- 没测试 → 至少手动跑一次最小 happy path
- 只看代码看起来对 ≠ 工作 —— 必须有"跑通"的证据

禁止"我觉得应该工作了"的交付。代码没跑过 = 没工作。

## 4. 实证 > 推测

讲"为什么 X"时必须有来源：

- 某函数行为怪 → 看源码 / 跑测试 / 读文档，不凭命名猜
- API 错误 → 看错误码 / 响应原文，不说"可能是服务器问题"
- 没验证的说法标"未验证"，不当事实给

"可能是 / 多半是 / 大概率" 在技术讨论里等同于猜测。直接说"真不知道，建议排查 Y" 比用柔化词糊过去更负责。
